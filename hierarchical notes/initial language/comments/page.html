  • Documentation
    • Documentation comments should be able to use variables defined in enclosing scopes by possible static if and foreach control structures. This is useful when defining multiple functions in a loop.
    • A documentation generator should be able to include the deduced return type of a function in the documentation.
      • This is because a keyword like C++'s 'auto' doesn't mean anything to the consumer of the documentation who doesn't have access to the function body.
      • The documentation generator could deduce the function's return type itself (exactly under the circumstances the compiler would).
      • Or better yet, the compiler could include the deduced type in the compiled module as part of the function's interface and the documentation generator, could read the information from there.
    • Tags
      • A documentation generator that reads documentation comments should support tags that correspond to sections of cplusplus.com/reference container pages like complexity, iterator validity, data races and exception safety (basic, strong, no-throw).
    • Ideas
      • Allow programmatic access to documentation comments?
      • Make documentation a constituent of function definition (similar to =test, =precondition, etc.) instead of a comment.
      • Support "documented unit tests" like those in D.
